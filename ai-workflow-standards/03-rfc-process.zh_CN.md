# 03 · RFC 流程

> **写代码之前先想清楚**。RFC（Request for Comments）就是把脑子里的设计落到纸上，让设计错误在 5 分钟的写作中被发现，而不是在 5 小时的编码后。
> 配套英文版：`03-rfc-process.en.md`。
> 模板：`templates/rfc-template.md` 和 `templates/design-spec-template.md`。

---

## 1. 为什么要写 RFC（即使一个人开发）

不写 RFC 的代价：
- 写到一半发现方向错了，回滚半天工作。
- 改完发现破坏了某个没意识到的契约。
- 三个月后回来不记得自己为什么这么设计。
- AI 助手没有方向，写出来的代码东拼西凑。

写 RFC 的收益：
- **强迫想清楚**：写下"为什么这么设计"时，逻辑漏洞会暴露。
- **AI 协作锚点**：RFC 是给 AI 助手的最强上下文，能显著降低跑偏概率。
- **未来可追溯**：`git blame` 只能告诉你"改了什么"，RFC 告诉你"为什么改"。
- **橡皮鸭审阅**：哪怕一个人开发，把设计写出来读一遍，比脑子里转靠谱。

## 2. 什么时候必须写 RFC

**触发条件**（满足任一即写）：

- 新增模块、新增子系统、新增公开 API。
- 改变现有公开 API 行为或默认值。
- 新增 middleware / interceptor / plugin 类的横切逻辑。
- 引入新依赖、新框架、新工具链。
- 跨多个文件（≥5）或多个模块的非平凡改动。
- 涉及数据 schema 变更、迁移、回滚策略。
- 涉及安全、认证、授权、加密相关逻辑。
- 性能优化且改动核心路径——必须先量化预期收益。
- 任何"以后很难改回来"的决策。

**不需要写 RFC 的场景**：
- Bug 修复（走 `05-bug-fix-workflow.zh_CN.md`）。
- 纯文档更新。
- 测试补充。
- Lint 警告修复。
- 单文件内的局部重构（不跨模块）。
- 配置项数值微调。

## 3. 两档 RFC

### 档位 A：轻量 RFC

适合：单模块、单关注点、改动范围可控。

文件位置：`docs/rfc-<short-name>.md`

内容（参考 `templates/rfc-template.md`）：
1. **问题**：当前痛点是什么。
2. **设计原则**：3-5 条不可妥协的约束。
3. **方案**：API 设计 + 使用示例。
4. **备选方案**：考虑过的其他做法及为什么放弃。
5. **迁移路径**：对现有代码/用户的影响。
6. **设计决议表**：每个关键决策的一行理由。

篇幅：100-300 行。

### 档位 B：完整 Spec + Plan

适合：跨子系统、影响深远、需要分阶段实施。

**两份文件**：
- **Spec**：`docs/specs/YYYY-MM-DD-<name>-design.md` —— 回答"**为什么这样设计**"。
- **Plan**：`docs/plans/YYYY-MM-DD-<name>.md` —— 回答"**按什么顺序改哪些文件**"。

#### Spec 内容

参考 `templates/design-spec-template.md`：

1. **目标**：要解决什么。
2. **调研发现**：现有代码/数据/日志的实际状态（带证据）。
3. **方案候选**：A/B/C 至少三个，对比利弊。
4. **推荐方案**：选哪个，理由。
5. **风险评估**：识别可能失败的地方。
6. **验证**：怎么证明方案可行（最小可复现脚本、原型代码、数据测试）。
7. **不做的事**：明确出界内容。

篇幅：200-500 行。

#### Plan 内容

参考 `templates/rfc-template.md` 中的 Plan 段：

1. **文件结构表**：每个文件操作类型（Modify / Create / Delete）+ 职责。
2. **Task 拆分**：每个 Task 一组改动，独立可验证。
3. **每个 Task 的 Steps**：勾选式清单。
4. **每 Task 的约束**：哪些不能改、哪些必须保留。

篇幅：每个 Task 30-100 行。

## 4. RFC 工作流

```
想法浮现
  ↓
起草 RFC（档位 A 或 B）
  ↓
自审：假装是别人写的，找漏洞
  ↓
（如果是团队）发给同事 review；如果是独立开发，让 AI 助手以"反对者"身份挑刺
  ↓
定稿：在文件头部写 Status: Approved
  ↓
进入实现（04-feature-workflow.zh_CN.md）
  ↓
实现过程中发现设计问题 → 回来改 RFC，记录"为什么改"
  ↓
实现完成 → PR 中链接到 RFC
```

## 5. 让 AI 助手审 RFC

独立开发的"peer review"靠 AI 助手。让 AI 玩两个角色：

### 角色 1：魔鬼代言人

```
读完这份 RFC，以最严苛的资深工程师视角回答：
1. 哪些假设没说清楚？
2. 哪些边界条件没考虑？
3. 这个设计在 10x 流量 / 10x 数据量下会怎么崩？
4. 备选方案里的 X 我没选，你能不能说服我反过来选它？
5. 用一段话总结这份 RFC，看我说的和你理解的是不是一回事。
```

### 角色 2：实现预测者

```
假设按这份 RFC 实现，列出：
1. 最容易出 bug 的 3 个地方。
2. 最难测试的 3 个地方。
3. 最可能返工的 3 个决策。
```

把 AI 的反馈叠回 RFC，**留痕**（在 RFC 末尾加 "Revision" 节记录重大修改）。

## 6. RFC 与代码的一致性

- RFC 是"设计意图"，代码是"实现现实"。
- 实现过程中如果偏离了 RFC → **必须**回来改 RFC，不能让 RFC 变成谎言。
- RFC 的 `Status` 字段生命周期：`Draft` → `Approved` → `Implemented`（或 `Superseded by <new-rfc>`）。
- 被废弃的 RFC 不要删——标记 `Status: Superseded`，链接到新的，保留历史。

## 7. RFC 索引

在 `docs/README.md` 或 `docs/rfc-index.md` 维护一张表：

```markdown
| RFC | 状态 | 日期 | 一句话目的 |
|-----|------|------|------------|
| rfc-create-agent-factory | Implemented | 2026-05-20 | 抽出纯参数的 SDK 工厂 API |
| rfc-extract-shared-modules | Approved | 2026-06-01 | 把 core 拆成可发布包 |
| rfc-grep-glob-tools | Draft | 2026-06-15 | 给 sandbox 加 grep/glob 工具 |
```

新加 RFC 立刻更新索引，不要等。

## 8. 何时跳过 RFC（且心安理得）

紧急 hotfix → 先修、后补 RFC（如果改了架构）。
原型探索 → 标记 `Status: Exploratory`，不进 index，丢弃也无所谓。

跳过的代价要在 PR 描述里写清楚："这个改动我没写 RFC，因为 X；如果后续发现 Y，再补。"

---

**一句话总结**：RFC 不是文书工作，是**最便宜的发现设计错误的方式**。写代码前花 30 分钟写 RFC，比写完 3 小时返工划算得多。
